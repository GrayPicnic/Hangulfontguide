Five ways Korean type differs from Latin한글 타이포그래피가 라틴과 다른 5가지

Sections are numbered 가·나·다 — Korean's A·B·C.섹션 번호는 가·나·다. 한국식 A·B·C입니다.

Skim these five points and every font decision downstream gets easier. Each one changes something practical: your file size, your UI layout, or your QA checklist.이 다섯 가지만 알아두면 폰트 고르기가 한결 쉬워집니다. 하나하나가 파일 용량, UI 레이아웃, QA 체크리스트에 곧바로 영향을 주는 항목들입니다.

1 · One font file can hold 11,172 syllables1 · 폰트 파일 하나에 최대 11,172자

Latin fonts cover ~230 glyphs. Korean letters combine into square syllable blocks, and every combination is a separate glyph. Most fonts ship the KS X 1001 standard set of 2,350 syllables; full coverage is 11,172. That's why a Korean font is measured in megabytes, not kilobytes — and why it lands directly on your build size and load time.라틴 폰트는 글리프 230개 정도면 충분합니다. 한글은 자모가 조합되어 네모난 음절 블록이 되고, 그 조합 하나하나가 전부 별개의 글리프입니다. 대부분의 폰트는 KS X 1001 표준인 2,350자를 담고 있고, 조합 가능한 전체 글자 수는 11,172자에 이릅니다. 한글 폰트 용량이 KB가 아니라 MB 단위인 이유이자, 빌드 용량과 로딩 시간에 직접 영향을 주는 이유입니다.

2 · Blocks on a grid, not letters on a line2 · 선 위의 글자가 아니라, 격자 위의 블록

Latin letters flow side by side along a baseline. Hangul syllables are designed inside an invisible square, like CJK characters. Kerning between pairs matters far less than in Latin — what matters instead is how a face handles stroke density, because some syllables pack far more strokes into the same square than others.라틴 문자는 베이스라인을 따라 옆으로 흐르지만, 한글 음절은 한자나 가나처럼 보이지 않는 정사각형 틀 안에 그려집니다. 그래서 글자 사이 커닝은 라틴만큼 중요하지 않습니다. 대신 획 밀도가 관건입니다. 같은 네모 안에 획이 훨씬 많이 들어가는 글자들이 있기 때문입니다.

3 · Bold gets dense, fast3 · 볼드는 순식간에 빽빽해집니다

Because strokes multiply inside a fixed square, heavy weights fill in much faster than Latin bolds. A weight that looks punchy in English can look clogged in Korean at the same size. Rule of thumb: pick your Korean weight one step lighter than your Latin instinct, then compare on screen.정해진 네모 안에서 획만 늘어나다 보니, 한글의 굵은 웨이트는 라틴 볼드보다 훨씬 빨리 빽빽해집니다. 영문에서는 시원하던 굵기가 같은 크기의 한글에서는 답답해 보일 수 있습니다. 요령은 하나입니다. 영문 기준보다 한 단계 가는 웨이트를 고른 뒤, 화면에서 직접 비교하세요.

4 · Baselines don't agree4 · 베이스라인이 서로 맞지 않습니다

Hangul is drawn to its square grid; Latin hangs from a baseline with ascenders and descenders. Mix a Korean face with a Latin face and the Latin usually looks small, high, or loose next to it. Fix it per pair: nudge the Latin size up ~5–10% or adjust line-height and vertical offset until mixed strings sit evenly.한글은 정사각 격자를 기준으로, 라틴은 어센더·디센더가 있는 베이스라인을 기준으로 그려집니다. 그래서 둘을 섞으면 라틴 쪽이 작아 보이거나 떠 보이기 쉽습니다. 해결은 조합마다 따로 해야 합니다. 라틴 크기를 5–10% 키우거나, 섞인 문장이 고르게 앉을 때까지 행간과 수직 위치를 조정하세요.

"Free" is not the same as "free to ship"'무료'와 '출시 가능'은 다릅니다

Korea has an unusually rich free-font culture — companies release fonts as brand marketing. Most allow commercial use, but the terms differ on the one thing games need most: embedding the font file inside your build.한국은 무료 폰트 문화가 유난히 풍성합니다. 기업들이 브랜드 마케팅 삼아 폰트를 배포하기 때문이죠. 대부분 상업적 이용은 허용하지만, 게임에 가장 중요한 조건 하나에서 갈립니다. 바로 폰트 파일을 빌드에 임베딩할 수 있는가입니다.

CAUTION주의

The game-dev trap: some Korean free-font licenses allow commercial use in images and video but restrict redistributing the font file itself — which is exactly what shipping it inside a game build does. Fonts under the SIL Open Font License (OFL) are the safe path: embedding, modification, and subsetting are all permitted. For anything else, read the distributor's page — some "free" fonts also prohibit reselling the font, using it in trademarks/logos without notice, or bundling it in font menus of paid software.게임 개발자가 빠지기 쉬운 함정: 일부 한국 무료 폰트는 이미지·영상에서의 상업적 이용은 허용하면서, 폰트 파일 자체의 재배포는 제한합니다. 그런데 게임 빌드에 폰트를 담아 배포하는 것이 바로 그 재배포에 해당합니다. SIL Open Font License(OFL) 폰트를 고르는 것이 가장 안전합니다. 임베딩, 수정, 서브셋팅이 모두 허용되니까요. 그 밖의 폰트는 배포처 페이지를 꼭 읽어보세요. 일부 '무료' 폰트는 폰트 재판매, 무단 로고 사용, 유료 소프트웨어의 폰트 메뉴 포함을 금지하기도 합니다.

Shipping Korean text without pain게임에 한글을 넣을 때 꼭 알아야 할 4가지

Four things that save builds: subsetting, engine setup, the line-height fix, and a proper QA string.빌드를 지켜주는 네 가지를 정리했습니다. 서브셋팅, 엔진 설정, 행간 조정, 그리고 제대로 된 QA 문자열입니다.

Subset the font — it's the biggest win서브셋팅 — 가장 확실한 용량 절감

You almost never need all 11,172 syllables. Subsetting to the KS X 1001 set of 2,350 (or just the glyphs your localization actually uses) can cut a multi-megabyte font dramatically. With fonttools installed:11,172자가 전부 필요한 경우는 거의 없습니다. KS X 1001의 2,350자(또는 로컬라이제이션에 실제로 쓰이는 글자만)로 서브셋팅하면 수 MB짜리 폰트를 극적으로 줄일 수 있습니다. fonttools 설치 후:

pyftsubset NotoSansKR-Regular.ttf \
  --unicodes="U+AC00-D7A3,U+1100-11FF,U+3130-318F,U+0020-007E" \

Engine notes: atlases will bite엔진 노트: 아틀라스를 조심하세요

Unity (TextMeshPro): a static atlas holding 2,350 syllables is huge — use dynamic atlas mode, or bake a static atlas from your localization's actual character set.

Unreal: the same logic applies to font caching; prefer runtime-cached faces over pre-baked textures for CJK.

Godot 4: dynamic fonts handle Hangul well by default; just watch the fallback chain so mixed KR/EN strings don't switch faces mid-line.

Web builds (WebGL/itch.io): load the subset .woff2 yourself and wait on document.fonts.ready before drawing UI — canvas-based renderers grab whatever font is loaded first and won't re-flow. iOS Safari in particular is quick to fall back to a system CJK font if your custom face hasn't finished loading in time.Unity (TextMeshPro): 2,350자를 담은 스태틱 아틀라스는 어마어마하게 큽니다. 다이내믹 아틀라스 모드를 쓰거나, 로컬라이제이션에 실제로 쓰인 문자만으로 스태틱 아틀라스를 구우세요.

Unreal: 폰트 캐싱도 같은 논리입니다. CJK는 미리 구운 텍스처보다 런타임 캐싱을 권장합니다.

Godot 4: 다이내믹 폰트가 기본적으로 한글을 잘 처리합니다. 다만 한·영 혼용 문장에서 줄 중간에 서체가 바뀌지 않도록 폴백 체인만 확인하세요.

웹 빌드(WebGL/itch.io): 서브셋한 .woff2를 직접 로드하고, UI를 그리기 전에 document.fonts.ready를 기다리세요. 캔버스 기반 렌더러는 먼저 로드된 폰트를 그대로 써버리고 다시 그리지 않습니다. 특히 iOS Safari는 커스텀 폰트 로딩이 늦으면 금방 시스템 CJK 폰트로 대체해버립니다.

The mixed-line fix한·영 혼용 줄 해결법

When Korean UI text contains Latin (item names, numbers, "HP"), set line-height on the container, not the font default — Korean squares make lines taller than Latin-only text expects. Start around line-height: 1.6 for body text and re-tune per face. If the Latin looks undersized next to Hangul, scale it up 5–10% rather than swapping fonts.한국어 UI에 영문(아이템 이름, 숫자, "HP")이 섞일 때는 폰트 기본값 대신 컨테이너에 행간을 직접 지정하세요. 한글은 네모꼴이라 영문만 있을 때보다 줄이 높아지기 때문입니다. 본문은 line-height: 1.6 근처에서 시작해 서체마다 다시 맞추면 됩니다. 한글 옆에서 영문이 작아 보인다면 폰트를 바꾸기보다 5–10% 키우는 편이 낫습니다.

One more line-wrapping gotcha for HTML/WebView UI: Korean breaks at spaces between words, but the default CSS line-breaking rules treat it like Chinese/Japanese and can split mid-word inside narrow boxes. Add word-break: keep-all; to Korean text containers so it wraps at word boundaries the way it actually reads.HTML/WebView UI라면 줄바꿈에서 하나 더 조심할 게 있습니다. 한글은 어절(단어) 단위로 띄어 쓰지만, 기본 CSS 줄바꿈 규칙은 중국어·일본어처럼 취급해서 좁은 박스 안에서 어절 중간이 잘릴 수 있습니다. 한글 텍스트 컨테이너에 word-break: keep-all;을 지정해 실제로 읽히는 단위로 줄이 바뀌게 하세요.

Latin fallback chain라틴 폰트 폴백 체인

Most free Korean fonts ship thin or no Latin glyph coverage, so pair each Korean face with a Latin font you actually like and let the browser/engine pick per character. In CSS, list the Latin face first so ASCII renders in it, then the Korean face for everything else:무료 한글 폰트는 라틴 글리프가 부실하거나 아예 없는 경우가 많습니다. 그러니 한글 서체마다 마음에 드는 라틴 폰트를 짝지어주고, 문자 단위로 알아서 골라 쓰게 하세요. CSS에서는 라틴 폰트를 먼저 적어 ASCII는 그 폰트로 렌더링되게 하고, 그 뒤에 한글 폰트를 이어줍니다:

font-family: "Archivo", "Pretendard",
  "Noto Sans KR", sans-serif;

Same idea in Unity/Unreal/Godot: set the Latin font as primary and the Korean font as its fallback asset, rather than trying to find one font that does both well.Unity/Unreal/Godot도 원리는 같습니다. 하나의 폰트로 둘 다 해결하려 하지 말고, 라틴 폰트를 기본으로 두고 한글 폰트를 폴백 에셋으로 연결하세요.
